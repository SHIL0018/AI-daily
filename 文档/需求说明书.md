## 1.1 当前实现对齐（截至 2026-07-03）

当前代码已经实现的 MVP 范围如下，本文档中涉及周报、月报、PDF、项目识别、系统级 Key、Key 加密、HTTPS 反向代理等内容仍属于后续扩展，除非本节明确列出，否则不视为当前已实现。

1. Windows 11 本地客户端位于 `desktop-client/`，使用 Electron + React + TypeScript + better-sqlite3。
2. 本地模型默认通过 Transformers / OpenAI-compatible 服务访问，默认地址 `http://127.0.0.1:8001/v1`，模型目录为 `desktop-client/local-models/ollama/Qwen3.5-0.8B`。
3. `start-client.cmd` 会检查 Node.js、npm、依赖、本地模型目录、本地模型服务，启动成功后命令行窗口自动退出；客户端关闭窗口时可选择后台运行或直接退出。
4. 远程服务端当前使用 FastAPI + Python `sqlite3` + SQLite 文件数据库，由 Uvicorn/systemd 在 Ubuntu 上运行；当前未接入 PostgreSQL、SQLAlchemy、Alembic、Redis 或 Celery。
5. Web 前端当前是 FastAPI 挂载的静态 HTML/CSS/JavaScript，已实现首页、记录管理、API 管理三视图和顶部/侧边导航。
6. DeepSeek API Key 当前由用户在 Web 的 API 管理页自行保存；未保存 Key 时点击 AI 分析会提示先配置 Key，后端返回 `API_KEY_REQUIRED`。当前没有 demo 账号，也没有服务端默认 Key fallback。
7. 当前 AI 分析只实现日报级 `daily` 分析；周报、月报和 PDF 导出仍是后续扩展。
8. 当前 Key 保存在服务端 SQLite 的 `user_api_keys` 表中，接口不回传明文，只返回脱敏 hint；生产环境加密存储仍是后续安全增强。

## 2. 项目背景

用户需要一个可以记录自己一天电脑活动的软件。软件需要在本地监控屏幕，定期采集当前屏幕、应用名称、窗口标题等信息，并调用本地轻量模型 Qwen3.5-0.8B 生成简短活动摘要。

客户端生成的活动记录会上传到远程服务器。远程服务器负责存储活动时间线，并通过 Web 页面渲染日报。由于日报、周报和深度复盘需要更强的归纳、分析和长上下文能力，Web 端的 AI 分析能力应通过后端代理接入 DeepSeek V4 API 实现。

系统整体采用“两级模型架构”：

1. 本地模型 Qwen3.5-0.8B：负责实时、轻量、隐私优先的屏幕活动短摘要。
2. DeepSeek V4 API：负责日报级、周报级、月报级的深度总结、复盘建议、趋势分析和结构化分析结果生成。

本系统面向用户本人使用，不用于隐藏式监控、员工监视、绕过系统权限或未经授权的数据采集。

## 3. 总体目标

### 3.1 核心目标

1. 在用户本地设备上透明地记录屏幕活动。
2. 使用本地 Qwen3.5-0.8B 生成短活动摘要。
3. 将时间段、摘要、类别、应用名称等结构化数据上传到远程服务器。
4. 服务端保存活动记录，并生成基础日报。
5. Web 端提供 DeepSeek V4 AI 分析入口。
6. DeepSeek V4 通过后端代理调用，不直接暴露 API Key 给浏览器。
7. 用户可以查看、编辑、删除、重新生成日报。
8. 用户可以导出 Markdown、JSON、CSV，后续支持 PDF。
9. 默认不上传原始截图、OCR 原文、键盘输入、鼠标轨迹、音频、摄像头内容。

### 3.2 重要变化

相比原始需求说明书，本版本增加以下需求：

1. 增加云端 AI 分析模块。
2. 增加 DeepSeek V4 API 接入需求。
3. 增加 AI 分析任务表、状态管理和结果缓存需求。
4. 增加用户对云端 AI 分析的确认与开关。
5. 增加用户自带 DeepSeek API Key 管理需求；当前 MVP 保存在服务端 SQLite，生产环境应加密。
6. 增加 Web 端“AI 分析”“重新分析”“分析模型选择”功能。
7. 增加日报深度分析、周报分析、趋势分析、复盘建议等功能需求。
8. 增加向第三方模型发送数据前的脱敏和最小化上下文要求。

## 4. 模型职责边界

## 4.1 本地 Qwen3.5-0.8B 职责

本地模型用于实时屏幕摘要。它的输入来自本地客户端，包括截图、应用名称、窗口标题、可选 OCR 文本和最近上下文。

主要职责：

1. 当前屏幕内容简短摘要。
2. 活动类别判断。
3. 是否疑似隐私内容判断。
4. 输出结构化 JSON。
5. 辅助生成活动片段。

不建议承担：

1. 长篇日报生成。
2. 多小时全局趋势分析。
3. 复杂复盘建议。
4. 周报、月报深度归纳。
5. 复杂项目识别。
6. 精确 OCR 替代。

---

## 4.2 DeepSeek V4 API 职责

DeepSeek V4 API 用于 Web 端和服务端的深度分析。

主要职责：

1. 生成自然语言日报。
2. 生成今日重点工作。
3. 识别连续专注时间段。
4. 识别频繁切换和碎片化时间。
5. 生成分类耗时解释。
6. 生成应用使用分析。
7. 生成复盘建议。
8. 生成周报、月报。
9. 根据用户编辑后的活动记录重新分析。
10. 将大量活动片段归纳为更可读的结构化报告。

不应承担：

1. 直接接收原始截图。
2. 直接接收 OCR 原文。
3. 直接接收密码、验证码、密钥等敏感信息。
4. 在浏览器端直接暴露 API Key。
5. 替代用户对隐私和上传范围的控制。

## 7. 核心业务流程

## 7.1 本地记录流程

1. 用户打开客户端。
2. 客户端检查屏幕录制权限。
3. 客户端检查本地模型服务是否可用。
4. 用户点击“开始记录”。
5. 客户端按配置采集屏幕与窗口信息。
6. 隐私过滤模块判断是否允许分析。
7. 如果命中隐私规则，生成“隐私内容，已跳过分析”记录。
8. 如果未命中隐私规则，调用 Qwen3.5-0.8B 生成摘要。
9. 客户端将活动记录写入 SQLite。
10. 同步队列将待上传记录发送到服务端。

---

## 7.2 服务端基础日报流程

1. 服务端接收活动记录。
2. 校验用户、设备和时间范围。
3. 进行幂等去重。
4. 保存到 `activity_records` 表。
5. 标记对应日期日报为待更新。
6. 使用规则算法生成基础日报。
7. Web 页面可展示基础日报。

---

## 7.3 Web 端 DeepSeek V4 AI 分析流程

1. 用户进入 Web 日报页面。
2. 页面展示基础日报。
3. 用户点击“AI 分析”或系统自动触发分析。
4. 前端调用后端接口：`POST /api/v1/ai-analysis/daily/{date}`。
5. 后端查询当天活动记录。
6. 后端进行数据脱敏和上下文压缩。
7. 后端根据用户设置选择 `deepseek-v4-flash` 或 `deepseek-v4-pro`。
8. 后端调用 DeepSeek V4 API。
9. DeepSeek 返回结构化分析结果。
10. 后端保存分析结果到数据库。
11. Web 页面展示 AI 日报、重点事项、复盘建议和趋势分析。

# 8.1 本地客户端功能

## FR-C-001 客户端启动

| 字段 | 内容 |

| 描述 | 用户启动客户端后，加载配置、检查权限、检查本地模型状态 |

5. 显示服务端同步状态。
6. 显示当前是否处于记录、暂停、停止、错误状态。

---

## FR-C-002 屏幕采集

| 字段 | 内容 |
|---|---|
| 优先级 | P0 |
| 描述 | 客户端定期采集当前屏幕，用于本地模型分析 |
| 验收标准 | 能稳定采集屏幕，且默认不上传原始截图 |

详细要求：

1. 支持单屏和多屏。
2. 默认采集间隔为 30 秒。
3. 支持配置采集间隔。
4. 支持应用切换时触发采集。
5. 支持窗口标题变化时触发采集。
6. 支持用户从空闲恢复时触发采集。
7. 截图默认仅在内存中处理。
8. 默认不保存本地原始截图。
9. 默认不上传原始截图。

---

## FR-C-003 本地 Qwen3.5-0.8B 摘要

| 字段 | 内容 |
|---|---|
| 优先级 | P0 |
| 描述 | 客户端调用本地 Qwen3.5-0.8B 生成简短活动摘要 |
| 验收标准 | 每条活动记录包含 summary、category、confidence、privacy_level |

详细要求：

1. 支持配置本地模型地址。
2. 支持 Ollama、本地 HTTP 服务、Transformers 服务。
3. 默认模型名称为 `qwen3.5:0.8b` 或 `Qwen/Qwen3.5-0.8B`。
4. 模型输出必须为 JSON。
5. 摘要长度建议为 10 到 40 个中文字符。
6. 摘要不能包含敏感原文。
7. 模型不可用时进入降级模式。
8. 模型超时时使用窗口标题或应用名称兜底。

---

## FR-C-004 本地隐私过滤

| 字段 | 内容 |
|---|---|
| 优先级 | P0 |
| 描述 | 在截图、模型调用和上传前执行隐私过滤 |
| 验收标准 | 命中隐私规则时不调用模型或不上传敏感字段 |

详细要求：

1. 支持应用黑名单。
2. 支持窗口标题关键词黑名单。
3. 支持隐私时间段。
4. 支持用户手动暂停。
5. 命中密码管理器、银行、支付、私密聊天等应用时不截图或只记录隐私时长。
6. 上传前再次执行摘要脱敏。
7. 默认不上传 OCR 原文。
8. 默认不上传窗口标题，或只上传脱敏后的窗口标题。

| 字段 | 内容 |

# 8.3 DeepSeek V4 AI 分析功能

## FR-AI-001 DeepSeek V4 API 接入

| 字段 | 内容 |
|---|---|
| 优先级 | P0 |
| 描述 | 服务端代理调用 DeepSeek V4 API，生成日报深度分析 |
| 验收标准 | Web 端能触发 AI 分析，后端能调用 DeepSeek 并保存结果 |

详细要求：

1. DeepSeek API Key 只能保存在服务端。
2. 浏览器前端不得直接调用 DeepSeek API。
3. 默认模型使用 `deepseek-v4-flash`。
4. 用户或管理员可配置使用 `deepseek-v4-pro`。
5. 支持 JSON 输出。
6. 支持超时控制。
7. 支持失败重试。
8. 支持分析结果缓存。
9. 支持用户手动重新分析。
10. 支持按日期、周、月生成分析任务。

# 12. 隐私与安全需求

## 12.1 用户确认

首次使用 DeepSeek V4 AI 分析前，系统必须提示用户：

```text
AI 深度分析会将你的活动摘要、时间段、类别和统计数据发送到 DeepSeek API。系统不会发送原始截图、OCR 原文、键盘输入、鼠标轨迹、音频或摄像头内容。你可以随时关闭 AI 分析功能。
```

用户确认后方可启用。

---

## 12.2 API Key 安全

1. DeepSeek API Key 只能提交到本系统后端，浏览器端不得持久化或回显明文。
2. 当前 MVP 由用户在 Web API 管理页自行配置，保存在服务端 SQLite 的 `user_api_keys` 表中。
3. 接口不得返回明文 Key，只返回配置状态和脱敏 `key_hint`。
4. Key 不得写入日志，不得提交到代码仓库。
5. 生产环境应加密保存或接入密钥管理服务；当前加密存储尚未实现。
6. 系统级默认 Key 已移除，当前无服务端默认 Key fallback。

---

## 12.3 DeepSeek 请求数据最小化

发送给 DeepSeek 的数据只包含：

1. 时间段。
2. 持续时长。
3. 活动摘要。
4. 活动类别。
5. 脱敏应用名。
6. 基础统计数据。
7. 用户手动备注。

默认不包含：

1. 原始截图。
2. OCR 原文。
3. 键盘输入。
4. 鼠标轨迹。
5. 摄像头。
6. 麦克风。
7. 原始窗口标题。
8. 聊天全文。
9. 文档全文。
10. 密码、验证码、密钥等敏感字段。

# 17. 总结

本版本需求说明书将系统明确升级为“两级 AI 分析架构”：本地 Qwen3.5-0.8B 负责实时屏幕摘要，DeepSeek V4 API 负责 Web 端日报、周报和深度复盘分析。

这种架构兼顾隐私、成本和分析能力：原始屏幕内容尽量留在本地，上传到服务器的是经过本地模型处理后的摘要和结构化元数据；需要高质量长文分析时，再由后端代理调用 DeepSeek V4 API。Web 端不直接接触 DeepSeek API Key，AI 分析请求经过后端脱敏、限流、缓存和状态管理。

最终系统应优先完成“本地记录 → 本地摘要 → 服务端存储 → 基础日报 → DeepSeek AI 分析 → Web 展示”的完整闭环，再逐步扩展周报、月报、项目识别、导出和个性化分析能力。
