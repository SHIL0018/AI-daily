# 后端接口设计（DeepSeek V4 API 分析版）

## 1. 文档目标

本文档描述屏幕活动记录与日报生成系统的后端 API 设计。后端负责用户认证、设备管理、活动记录接收、活动记录查询、规则版日报生成、DeepSeek V4 API 分析、日报查询、编辑删除和导出。

Web 端的“AI 分析”功能由后端代理调用 DeepSeek V4 API 实现。浏览器端不得直接保存或调用 DeepSeek API Key。后端需要统一完成权限校验、数据脱敏、prompt 组装、异步任务、失败重试、结果缓存和 token 成本统计。

后端设计核心原则：

1. 所有用户数据必须鉴权访问。
2. 客户端上传接口必须幂等。
3. 默认不接收原始截图。
4. 接口参数必须严格校验。
5. 用户只能访问自己的设备、记录和日报。
6. 活动记录变更后必须标记日报需要重新生成。

---

## 1.1 当前实现对齐（截至 2026-07-03）

当前后端实现位于 `server/app`：

1. API 框架为 FastAPI，数据库访问使用 Python 标准库 `sqlite3`，当前没有 SQLAlchemy/Alembic。
2. 数据库为 SQLite 文件数据库，启动时由 `init_db()` 自动建表。
3. AI 分析任务通过 FastAPI `BackgroundTasks` 执行，当前没有 Redis/Celery/RQ。
4. Web 静态资源由 FastAPI 挂载 `/static`，并提供 `/`、`/home`、`/records`、`/api-keys` 路由。
5. DeepSeek API Key 由每个用户通过 `/api/v1/api-keys/deepseek` 管理；无 Key 时创建 AI 分析任务返回 `API_KEY_REQUIRED`。
6. 当前只支持日报 AI 分析，`analysis_type` 仅接受 `daily`。

### 3.1 Base URL

```text
/api/v1
```

### 3.2 认证方式

除注册、登录、刷新 Token 外，所有接口需要：

```http
Authorization: Bearer <access_token>
```

### 3.3 请求格式

```http
Content-Type: application/json
```

### 3.4 通用成功响应

```json
{
  "success": true,
  "data": {}
}
```

### 3.5 通用失败响应

当前 FastAPI 接口失败时使用 `HTTPException(detail={...})`，响应结构如下：

```json
{
  "detail": {
    "code": "INVALID_PARAMS",
    "message": "参数错误"
  }
}
```

## 4. 错误码设计

| 错误码 | HTTP 状态码 | 说明 |
|---|---:|---|
| INVALID_PARAMS | 400 | 参数错误 |
| UNAUTHORIZED | 401 | 未登录或 Token 无效 |
| TOKEN_EXPIRED | 401 | Token 已过期 |
| FORBIDDEN | 403 | 无权限访问 |
| NOT_FOUND | 404 | 资源不存在 |
| PAYLOAD_TOO_LARGE | 413 | 请求体过大 |
| RATE_LIMITED | 429 | 请求过于频繁 |
| DUPLICATED_RECORD | 200 / 409 | 记录重复，幂等处理 |
| INTERNAL_ERROR | 500 | 服务端异常 |
| SERVICE_UNAVAILABLE | 503 | 服务不可用 |
| API_KEY_REQUIRED | 400 | 用户尚未配置 DeepSeek API Key |

幂等规则：


## 9. 日报生成服务

### 9.1 触发条件

1. 新活动记录上传。
2. 活动记录编辑。
3. 活动记录删除。
4. 用户打开日报但日报不存在。
5. 用户手动重新生成。
6. 每日定时任务。

### 9.2 生成流程

1. 查询当前用户当前日期的活动记录。
2. 过滤 is_deleted = true 的记录。
3. 按 start_time 排序。
4. 合并相邻相似活动。
5. 统计总时长、有效时长、空闲时长、隐私时长。
6. 统计 category_stats。
7. 统计 app_stats。
8. 提取 highlights。
9. 生成 title 和 overview。
10. 写入 daily_reports。

## 10. DeepSeek V4 AI 分析接口

### 10.0 用户 DeepSeek API Key 管理接口（当前已实现）

当前 Web 端左侧导航提供“API 管理”，用户必须自行保存 DeepSeek API Key 后才能发起 AI 分析。后端不返回明文 Key，只返回配置状态和脱敏提示。

查询状态：

```http

Authorization: Bearer <access_token>
```

响应：

```json
{
  "success": true,
  "data": {
    "provider": "deepseek",
    "configured": true,
    "key_hint": "sk-xxx...abcd",
    "updated_at": "2026-07-03T12:00:00+08:00"
  }
}
```

保存或更新：

```http
PUT /api/v1/api-keys/deepseek
Authorization: Bearer <access_token>
Content-Type: application/json
```

```json
{
  "api_key": "sk-..."
}
```

删除：

```http
DELETE /api/v1/api-keys/deepseek
Authorization: Bearer <access_token>
```

未配置 Key 时调用 `POST /api/v1/daily-reports/{date}/ai-analysis` 会返回：

```json
{
  "detail": {
    "code": "API_KEY_REQUIRED",
    "message": "请先在 API 管理中配置 DeepSeek API Key"
  }
}
```

### 10.1 设计原则

1. Web 前端只能调用本系统后端接口，不能直连 DeepSeek API。
2. DeepSeek API Key 由用户在 API 管理页配置，保存在服务端 `user_api_keys` 表；当前不提供服务端默认 Key。
3. 后端调用 DeepSeek 前必须对活动记录进行二次脱敏和压缩。
4. DeepSeek 分析任务必须异步执行，避免阻塞日报页面。
5. 规则版日报必须始终可用，DeepSeek 分析失败时自动降级。
6. 同一用户、同一日期、同一输入 hash 的 AI 分析结果应缓存复用。

### 10.5 服务端调用 DeepSeek V4 的内部请求

当前实现使用 `httpx` 调用 OpenAI-compatible `/chat/completions` 接口，并从当前用户保存的 Key 中读取 Authorization：

```python
with httpx.Client(timeout=settings.deepseek_timeout_seconds) as client:
    resp = client.post(
        f"{settings.deepseek_base_url.rstrip('/')}/chat/completions",
        headers={"Authorization": f"Bearer {user_api_key}"},
        json={
            "model": model_name,
            "messages": [
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_payload},
            ],
            "stream": False,
            "temperature": 0.2,
            "response_format": {"type": "json_object"},
        },
    )
    resp.raise_for_status()
```

### 10.6 DeepSeek 分析任务状态机

```text
pending
  ↓
running
  ↓
succeeded

失败路径：
running -> failed -> fallback

缓存路径：
pending -> succeeded(cached=true)
```

状态说明：

| 状态 | 说明 |
|---|---|
| pending | 任务已创建，等待执行 |
| running | 正在调用 DeepSeek V4 API |
| succeeded | 分析成功 |
| failed | API 调用或解析失败 |
| fallback | 使用规则版日报作为降级结果 |

### 10.7 DeepSeek 调用前脱敏校验

调用前必须检查 payload 中是否包含以下内容：

1. base64 图片。
2. OCR 全文。
3. 连续疑似身份证号、银行卡号、手机号。
4. Token、API Key、password、secret 等敏感字段。
5. 过长的聊天原文或文档原文。

若校验失败：

```json
{
  "detail": {
    "code": "AI_ANALYSIS_PRIVACY_CHECK_FAILED",
    "message": "AI 分析输入未通过隐私校验"
  }
}
```

### 10.8 限流与成本控制

建议规则：

| 限制项 | 默认值 |
|---|---:|
| 单用户每日 standard 分析次数 | 20 |
| 单用户每日 deep 分析次数 | 5 |
| 单次最大输入记录数 | 1000 |
| 单次最大输入 tokens | 根据模型窗口和成本配置 |
| DeepSeek 超时时间 | 120 秒 |
| 最大重试次数 | 2 |

## 11. 权限设计

每个接口都必须基于当前 user_id 校验资源归属。

| 资源 | 权限规则 |
|---|---|
| device | device.user_id == current_user.id |
| activity_record | record.user_id == current_user.id |
| daily_report | report.user_id == current_user.id |

禁止客户端传入 user_id 后直接信任。user_id 必须来自 Token。

---

## 12. 限流和大小限制

| 项目 | 限制 |
|---|---|
| 登录接口 | 每 IP 每分钟 10 次 |
| 批量上传 | 每次最多 100 条 |
| 请求体大小 | 默认不超过 1MB |
| 导出接口 | 每用户每分钟 10 次 |

---

## 13. 验收标准

1. 用户可以注册和登录。
2. 客户端可以注册设备。
3. 停用设备后不能上传记录。
4. 批量上传支持幂等去重。
5. 查询活动记录只返回当前用户数据。
6. 编辑记录后日报被标记 stale。
7. 删除记录后日报不再显示该记录。
8. 查询日报可以返回完整结构。
9. 重新生成日报可以正常执行。
10. 导出 Markdown 可用。
11. 服务端默认不接收原始截图。
12. 日志不记录 Token、密码和完整请求体。
